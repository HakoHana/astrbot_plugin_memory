# 架构对比分析

> 对比对象: `memory-analysis.md` (第三方记忆系统分析) vs `架构设计.md` (本插件)
> 分析角度: 各自优点 + 本插件的优化空间

---

## 一、定位差异

| 维度 | memory-analysis.md | 本插件 |
|------|-------------------|--------|
| 目标场景 | VUP/多平台/AI 数据流 | AstrBot QQ Bot |
| 分析对象 | Hermes / Claude-Mem / EverOS | 自己的插件 |
| 视角 | 外部评估，选型 | 内部设计，实现 |
| 粒度 | 粗，对比维度多 | 细，实现细节多 |

---

## 二、memory-analysis.md 的优点（值得我们学的）

### 1. 渐进式检索（Claude-Mem）

原文提出的 3 层检索：

```
search(query) → 返回 ID + 摘要（~50-100 token）
timeline(id)  → 获取周围时间窗口上下文
get_observations(ids) → 获取完整记录
```

**我们现在的做法**：FTS5 搜索 → 直接返回完整原子。没有分层，token 浪费。

**优化空间**：召回时先返回摘要级别结果，用户/LLM 需要详情时再展开。可以省 3~5 倍 token。

### 2. Skill 自进化（Hermes）

Hermes 能从任务中自动提炼可复用的 Skill，技能在使用中自改进。

**我们现在的做法**：没有技能系统，所有判断都走 LLM，成本高。

**优化空间**：可以做一个轻量级的「模式识别」— 如果某个原子类型 + 话题组合反复出现，自动提升重要度，减少 LLM 判断频率。

### 3. 周期性 Nudge（Hermes）

Agent 定期主动审视记忆，强化或修正。

**我们现在的做法**：只有被动触发（消息到达才整理），没有主动 consolidation。

**优化空间**：我们已经有了 `_decay_loop`，可以扩展为「定期审视」— 不仅仅是衰减，还做一致性检查和跨日记关联。

### 4. Markdown 作为 Source of Truth（EverOS）

EverOS 的核心设计：所有记忆以 `.md` 文件持久化，可读、可 git 版本化、可 Obsidian 打开。

**我们现在的做法**：content 字段已经是 frontmatter + Markdown 格式，但没有导出为独立 `.md` 文件。

**优化空间**：加一个导出功能，一键将所有日记导出为 Obsidian 仓库。

### 5. 多维检索（EverOS）

EverOS 支持按 `user_id`、`agent_id`、`project_id` 等多维度独立检索。

**我们现在的做法**：只能按 user_id 过滤。

**优化空间**：在 `diary_fact_links` 上加类型标签，可以实现按话题、情绪、重要性区间的组合过滤。

### 6. 混合检索（EverOS / Claude-Mem）

两者都支持 BM25 + 向量混合检索。

**我们现在的做法**：只有 FTS5（BM25），没有向量。

**优化空间**：低优先级。`memory_atoms` 表已经有 `embedding` 和 `embedding_model` 预留字段，数据量增大后可接入。

## 三、我们架构的优点（memory-analysis.md 没有的）

### 1. 去重强化机制

`atomic_facts` 的 content UNIQUE + 重复引用强化重要度，三个第三方系统都没有这个设计。

### 2. 图谱副产物

我们写日记时自动解析 `[[实体]]` 建图，零独立调用。Hermes / Claude-Mem / EverOS 都没有图谱关联。

### 3. 用户身份可追溯

`sender_name` 历史快照 vs `user_registry` 最新名字双轨制。EverOS 有 user.md 但只是单一版本。

### 4. 自描述内容格式

frontmatter + Markdown，导出即用。EverOS 有类似设计，但我们的字段更结构化。

### 5. 全局事实 + 日记关联的双层模型

```
原子事实 → 全局唯一（atomic_facts）
         → 日记关联（diary_fact_links）
         → 通过 user_id 溯源
```

这个三层的归属链路，三个系统都没有。

### 6. 全 SQLite 零依赖

EverOS 依赖 LanceDB + SQLite，Claude-Mem 依赖 Chroma + SQLite。我们只依赖 aiosqlite，部署成本极低。

## 四、优化空间汇总（按优先级）

| 优先级 | 优化点 | 参考来源 | 预估工作量 |
|--------|--------|---------|-----------|
| **P0** | **导出为 Obsidian 仓库** — 一键将所有日记导出为 `.md` 文件 | EverOS | 小（脚本级） |
| **P1** | **渐进式检索** — 搜索结果先返回摘要，再按需获取详情 | Claude-Mem | 中（修改 retriever） |
| **P2** | **主动审视 Nudge** — 定期审视记忆，跨日记关联 | Hermes | 中（扩展 _decay_loop） |
| **P3** | **多维过滤** — 按话题/情绪/重要度范围组合搜索 | EverOS | 中（API + 前端） |
| **P4** | **向量检索** — 接入 embedding，实现语义搜索 | EverOS / Claude-Mem | 大（依赖 + 索引） |
| **P5** | **Skill 模式学习** — 自动识别高频模式，减少 LLM 调用 | Hermes | 大（新模块） |

## 五、一句话结论

```
memory-analysis.md 给了宏观视野和参照系
架构设计.md 给了工程实现细节

我们现在的架构在「去重、图谱、身份溯源」上优于三者
在「渐进式检索、多维过滤、主动审视」上有提升空间
```
